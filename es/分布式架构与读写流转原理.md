# 分布式架构与读写流转原理

## es集群与分片

1. 主分片  ：解决了存不下的问题

   - 创建索引的时候 es会切块 比如3块
   - 存储到不同的服务器【Node 上 
   - 数据被分摊成3份 完美解决了单机容量瓶颈

2. 副本分片：解决了 怕宕机 和高并发读 问题

   - es会为每一个主分片 创建一个或多个副本分片
   - 高可用：副本分片不会和主分片放在一台服务器上 主分片宕机 副分片会瞬间成为主分片 系统正常运行
   - 高并发：大量search请求进来 主分片和副分片都可以提供search服务 提升吞吐量

   ## 路由机制

   发生写请求的时 服务器、分片很多 数据要写哪里呢？

   **路由公式：**

   shard-num=hash(_routing)%num_primary_shards

   - id先做hash
   - 再对主分片num取模

   所以es创建好分片后 主分片数量无法改变 否则路由公司全部失效 索引不可用 找不到数据了



## nrt 近实时与 落盘原理

数据写入:

1. 写入buffer 与 tanslog
   - 写入缓冲区 速度非常快 
   - 但是这条数据不可见 因为没有转换为倒排索引
   - 防丢失：为了防止断电后 buffer信息丢失 es会把这次操作追加写到disk上的translog文件 类比redolog 做崩溃恢复
2. refresh nrt的核心

信息一直在buffer不行 用户要着急搜索 

- es每隔1s 就会做一次refresh
- refresh 会把 buffer的数据 构建成mini倒排索引：segment
- segment不会写进disk 速度慢 会写入os cache
- 只要进入os cache 用户就可以查询到 
- 所以是近实时 有1s延迟

3. flush 落盘

translog越来越大 os cache 里 segment很多 

translog大到一定程度 或 默认30min 会触发flush

- flush 会强制把所有segment刷入disk fsync 文件同步操作
- 然后清空translog 这一步 数据才算落盘成功
