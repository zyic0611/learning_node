## 树化与退化

- 当链表长度达到8 尝试转化成红黑树

- 双重条件 只有数组长度达到64 才会转化为树 否则不如进行扩容 不用搞那么复杂的树
- 退化阈值 红黑树删除数据 节点掉到6的时候 就会退化  节点太少 树的维护成本(左旋右旋 ) 比链表高

单向链表可以解决哈希冲突  红黑树可以优化哈希冲突下的查询效率  on->logn

## 长度均为2的幂次方

不管初始制定什么长度 都会调用tableSizeFor 找一个2的幂次方大小。

为了优化哈希值的 取模运算:

举例子 如果长度刚好是16 

则就让除数是16-1=15 

发现15的二进制为 1111 高位全为0

则一个数对16取模 就等价于与16-1做与运算 。

与运算速度比%快几十倍

**`& (length - 1)` 等价于取模**

### 做与运算的问题 hash()扰动函数

比如一个数字有32位 则与长度为16的hashmap 计算哈希值 则高26位的信息会全部丢失。

为了解决这个问题  源码采用： (h = key.hashCode()) ^ (h >>> 16)

也就是先把h右移动16位 拿出高16位的信息  与低16位进行异或运算 再送进去求哈希值。

这样最后16位旧结合了高低两部分的信息。



## 多线程并发扩容机制的问题

HashMap扩容机制 默认扩容数组长度为2倍。Resize

### jdk1.7 死循环

1. jdk 1.7 采用头插法迁移 如果一个链表挂着是 a->b->c 则扩容迁移之后会变成 c->b->a。

2. 在多线程情况下 A刚准备迁移 他记得 a->b 

3. 此时A休眠了一会 B全部迁移结束 物理变成了b->a

4. A线程的规矩 需要进行头插法 则A线程直接用a->b 而b->a 则出现了死循环

5. 扩容核心方法

   ```transfer()```

   Entry next = e.next;      // 第一步：记住下一个包裹是谁 是b
   e.next = newTable[i];     // 第二步：把我手里的包裹，指向上一次放进去的包裹（头插法核心） 此时是B弄好的 b->a 所以 a->b->a
   newTable[i] = e;          // 第三步：把我手里的包裹，变成当前柜子的第一个（头）
   e = next;                 // 第四步：去拿下一个包裹

   

   所以产生锁扣的根本原因是 线程恢复后 其内部的局部变量e和next是旧值 而操作主内存中的数组头节点newTable[i]以及ing被其他线程更新了

### jdk 1.8 数据最多覆盖丢失

1. 使用尾插法 数据迁移 链表顺序不变
2. 两个线程同时迁移 最多产生数据覆盖丢失

## 扩容后哈希值的计算

jdk1.7前 扩容后 所有元素的哈希值 都要重新计算一遍



jdk1.8： 采用高低位链表拆分

因为扩容后就是多进行与运算的时候多了一位 

比如4位变成5位

- 如果该元素倒数第五位为0 则扩容后位置不变
- 如果为1 则位置变为原位置+老长度

而倒数第五位实际上就是老数组长度位。 所以源码使用e.hash & oldCap  与运算 去判断位置是否=0 因为oldCap刚好只有那个位置位1 其他位置均为0



## null值

key可以为null值 hashmap使用hash（）方法计算哈希值 当key为null时 hash值之前为0 不走hash方法

但是只支持null为key的有1个 也就是  value为null的可以有多个。 

这是由于hashmap本身就是key只能唯一的 所以key为null的也只能有一个 

但是存放在0号桶 可能也有别的key
