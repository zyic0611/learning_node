# 缓存三剑客

## 缓存击穿

### 单一热点key过期+突然过期

一个key qps特别高 热key在redis里一直扛着压力。

如果这个key突然在redis里消失，则全部的并发请求都越过redis 来到mysql，而mysql的抗并发能力很差，通常几千久挂了，则会把mysql的连接池打满，cpu飙升，数据库瘫痪。

### 解决策略

1. 互斥锁/分布式锁   

   - 原理：当很多请求 发现redis的缓存没有命中 不让他们去查数据库  而是让他们去抢锁 保证同一时间自由一个请求到达mysql 由于是分布式环境 所以可以抢redisson锁

   - 抢到锁的线程才有资格去查mysql 并且重新写会redis 其他抢锁失败的线程则休息几十ms 然后重新查redis

   - 优点：保证了数据绝对的一致性 查到的都是最新的数据

   - 缺点：加锁会导致其他线程排队阻塞 如果查sql过程慢 会导致整体吞吐量下降

2. logical expiration 
   - 原理：在redis里 热key永远不设置真正的ttl【物理不过期 在value里 人为加上一个逻辑过期的字段 exprie_time=... 
   -  流程：a线程查数据 发现时间字段 逻辑过期了 此时a会尝试去获取互斥锁 抢到锁后开启一个后台异步线程去查mysql 并更新redis 
   - a线程会直接返回旧数据 其他抢不到锁的线程也是直接返回旧数据
   - 优点：空间换时间 性能很高 感受到延迟
   - 缺点：牺牲了数据的一致性 看到旧数据

## 缓存雪崩

大面积key同时过期或者redis宕机  ，整个面都被打穿。

### 

1. 大面积key同一时间集中过期
   - 大量的key的过期时间相同 则在这在过期的瞬间 查询流量全部砸向mysql
   - 解法：采用过期时间打散 TTL Jitter 不用固定值设置ttl 在基础时间上加上随机数
   - 在随机数区间内就会陆续过期 把同时打向数据库的流量平摊薄了
2. redis节点宕机【物理毁灭
   - 物理机断电 或者大key导致oom redis服务直接失效
   - 解决方法 A：事先预防 建立高可用集群 使用 主从复制+哨兵机制 / redis cluster 一个节点挂了 选出新的主节点
   - 解法 B：事中抗压 限流或者降级  如果qps超过mysql极限 剩下的请求全部拦截 返回系统繁忙
   - 解法C：多级缓存兜底  热key中的本地缓存

缓存击穿会引起雪崩。

- 雪崩不仅是大面积key同时过期，mysql挂了也叫做雪崩，因为底层mysql死了，此时全部查不到数据，整个系统全面瘫痪
- 击穿会砸坏mysql 把连接池打满 导致雪崩。
- 一个热key 没做好互斥锁（防止击穿）或者限流兜底（保护mysql） 就会导致缓存雪崩。



## 缓存穿透

数据库没这个数据 redis也没有这个数据

场景： 疯狂请求系统接口 查询不存在的id redis没有 查mysql mysql也没有 返回空。由于返回空，redis也没写，下次再请求不存在的id ，请求还是打在了mysql上，redis‘完全没起作用，流量全部流向数据库。

### 缓存空对象

mysql返回空的时候 存入空结果在redis 并且设置一个极短的过期时间

但是黑客如果每次用不同的id来攻击 redis会被塞满无用空对象 浪费内存

### 布隆过滤器 bloom filter

判断一个数据存不存在 直接在java里用一个hashset存储所有ID不就行了吗？

不行 数据库有10亿1用户 则一个hashset就使用了几十GB的内存 oom

bloom filter解决了在海量数据下极快判断一个元素存在与否

1. 底层结构 采用 BitMap+多个Hash fuc
2. 工作原理 存入一个id 10086
   - 用所有hash fuc 对10086进行计算
   - 得出n个结果
   - 在bitmap里把n个位置全部改成1
3. 拦截恶意请求
   - 请求先经过bloomfileter
   - n个hashfuc计算恶意id
   - 检查bitmap 只要一个是0 则id不存在 
   - 直接拦截 不会进redis和mysql
4. 误判率
   - bitmap长度有限 存入数据越多 1就越多 一个id刚好算出来的位置 由于存入数据很多 这几个位置由于其他正常id的映射 就污染成1 则误判。
5. 核心法则：你不存在肯定不存在 说你存在 你不一定存在
6. 缺陷：无法删除 如果删除一个数据 会修改正常数据的状态 一般不支持删除操作。要删除大批量数据，通常采用定期通过定时任务 把全亮数据库读取一次 重建一个bloom filter
