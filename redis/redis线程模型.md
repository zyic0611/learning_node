# redis线程模型

## redis为什么快

1. 纯内存访问 数据都在内存中
2. 单线程核心逻辑： 避免了多线程频繁的上下文切换开销 彻底免去了 加锁 释放锁 的逻辑
3. 极度优化的数据结构 sds很快的查询长度 跳表 很快的范围查询 listpack 极致节省内存 从微观层面把性能压缩到了极致
4. i/o多路复用  利用系统底层的epoll机制 保证了即使是单线程 也可以高效处理并发网络连接

## 网络模型是什么

1. redis 6.0以前 单reactor
2. redis6.0以后 单reactor多线程模型 ：
   - 主线程 负责接受 连接 和 核心的命令执行
   - 耗CPU的**网络数据读写**与**协议解析**剥离给多个io线程
   - 将极其耗时的 Socket 读写和数据包解析分配给工作线程组，但核心的指令执行依然串行交由主线程处理，彻底打破了**网络 I/O** 的瓶颈。

## 如何实现I/O多路复用

1. 核心依赖linux底层epoll系统调用 mac有mac的kqueue
2. 同时监听所有服务器 也就是所有的Socket文件描述符(FD) 全部丢进epoll里监听
3. 只有当某个socket有数据可读或可写的时候 epoll才会通知redis的主线程处理。主线程不会主动询问 只会被动响应 实现了单线程高效管理多个并发连接

## redis那些地方使用了多线程

1. redis4.0后引入了后台多线程 (异步任务)
   - 也就是删除大key的时候 如果在主线程删除 会卡死 
   - redis引入unlink命令 删除大对象的时候 交给后台异步线程慢慢删除
2. redis 6.0后引入了I/O多线程
   - 为了解决网卡读写的CPU瓶颈 引入多线程 
   - 即使开启多线程 redis处理核心业务 依旧是单线程 所以写set get 不用考虑并发安全问题
