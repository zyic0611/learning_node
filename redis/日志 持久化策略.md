# 日志 持久化策略

redis是纯内存数据库 服务器断电或者宕机 数据会丢失 。持久化的目的是把内存里的数据存在硬盘上 以便重启时恢复。



## RDB redis database

内存快找 会生成一个紧凑的二进制压缩文件。dump.rdb

### 触发机制

1. save: 主线程执行 会阻塞所有客户端请求 直到rdb文件生成完毕 在生产环境禁用
2. bgsave  主线程会fork出一个子线程 由子线程生成rdb 不阻塞主线程 【默认方式

### cow机制

bgsave 期间 主线程还在接受写请求 则此时快照拍的会与主线程不一致 怎么保证数据一致性

使用copy on write   写时负责 子线程被fork出来的时候 与主线程共享一片物理内存

主线程接受读请求 无所谓

主线程接受写请求  要修改值。 os会把要被修改的这块数据copy副本出来 主线程去修改副本。 子线程仍然copy旧数据去rdb

结果是rdb 记录的是旧数据 

###  优缺点

- 优点： 恢复速度很快，因为是二进制文件 字节加载到内存即可  文件体积小 很适合做全亮灾备
- 缺点： 容易丢失数据 如果设置5 min 拍照一次，单机的时候 会丢失这5min的数据 fork 子线程 内存过大 会导致主线程短暂卡顿

## AOF  append only file 

追加日志 把redis执行过的每一条 "写" 命令 以文本日子的形式 追加到文件中 (appendonly.aof) 重启时 redis从头到尾执行一下这些命令 就可以恢复数据

### 刷盘策略 fsync

- always(同步写回) 每次执行完wirte命令 立刻同步写回磁盘 绝不丢失数据 但是效率极差
- Everysec 每秒写回 先把命令写到 系统内核缓冲区  每隔一秒由后台刷入磁盘。 【默认配置  推荐 只会丢失一秒的数据
- No redis只写入内核缓冲区 什么时候刷入磁盘 由os决定 性能最好 但是不安全

### aof 重写机制

- 对一个key执行 100次 incr 【自增  aof记录100条命令 但是其实只要记录最后的值 去set即可
- 原理 bgrewriteaof:
  - 主线程fork出一个bgrewriteaof子进程
  - 子进程扫描当前内存数据 生成一套最精简的写命令集 写入最新的aof文件
  - 重写期间 主线程收到的新写请求 会同时写入原aof缓冲区和aof重写缓冲区
  - 新aof文件生成后 会把重写缓冲区的增量命令追加到新文件中 最后替换掉老文件 

### 优缺点

- 优点 数据安全很高 最多丢失一秒数据 日志文件是追加模式 中途宕机也不容易损坏
- 缺点：恢复速度慢 重新执行很多命令 并且文件体积很大

## 混合持久化

redis4.0引入   rdb恢复数据块 容易丢数据 aof安全 但是慢 结合

1. 触发重写 ：redis触发了aof重写机制 bgrewriteaof 
2. fork子进程  此时【COW机制开始生效了】 此时不会使用aof重写机制 子线程的精简命令 
3. 子进程会直接拍摄rdb快照 写入new file中
4. 此时正在双写 在子进程写rdb文件的时间中 主线程还是会接受新的写操作
   - 只要是双写状态 就是基于COW 主线程要修改数据 触发os的cow 只能在副本上修改
   - 同时这些new write order 都会 放进aof重写缓冲区 类似于aof重写机制后半段
5. 子进程写完rdb数据后 主线程知道了 就会吧aof重写缓冲区中的增量命令 以aof 文件追加的格式 写入new file的后半段
6. 最后用混合文件替换磁盘里的老aof



## 触发

aof 日常存储(增量) 靠刷盘策略fsync 追加在aof文件

aof 文件瘦身(全量) 也就是触发aof重写机制  靠命令 手动bgrewriteaof 或者 自动触发 要去配置的 比如aof文件大小达到上线

rdb 触发 命令触发【sava bgsave 或者自动触发 也是去配置 自动触发是使用不阻塞的bgsave



## 三种关系

### 只会有两种持久化文件

1. dump.rdb
2. appendonly.aof 

如果使用了混合持久化 是存在aof文件里

### 三种配置组合

1. 纯rdb模式 默认配置 关闭aof
2. 纯aof 显示关闭混合持久化 系统虽然可以调用命令触发rdb 但是宕机恢复默认aof
3. 混合持久化 开启aof并且开启混合持久化 
   - **主力是 AOF（混合版）：** 所有的日常刷盘、自动重写，全都是围绕着这个混合版 AOF 进行的。宕机重启时，Redis 也**只会读取这一个文件**来恢复数据。
   - **辅助是 RDB：** 虽然重启不靠 RDB 文件了，但运维人员依然可以每天凌晨写个脚本，给 Redis 发送一个 `bgsave` 命令，生成一个纯粹的 `dump.rdb` 文件，把它扔到阿里云 OSS 或者其他冷备服务器上，用来做**灾难级的全量备份**。

所以混合持久化就是aof的一种功能增加，当场环境下会开启aof混合模式 并且定期rdb备份。只要开启aof redis重启默认优先加载aof
