# 大key&热key

## 大key

是value很大。

1. 标准
   - String value>10kb/5kb
   - 集合(hash set zset list) 元素数量超过5000个 or >10MB 
2. 危害
   - 阻塞单线程（核心 ： redis处理命令采用的是单线程 如果用HGETALL去读取一个很多元素的hash 或者del一个50MB的String 主线程会被卡住 此时别的客户端无法请求redis。
   - 网络拥堵 一个1mb的大key qps为1000的时候 每秒都会产生1gb的网络流量 打满网卡 拖死其他服务
   - 集群倾斜  在redis cluster中 大key只会落在某一个固定的节点上  会导致该节点内存远超其他节点 触发oom 或者在扩容缩容迁移data的时候容易falure
3. 排查大key
   - redis-cli --bbigkeys 
     - 官方自带 
     - 只能找出每种类型的top1 对于集合 他统计的是元素个数 不是内存大小
   - memory usage <key> 精准计算某个key的占用大小
   - **离线rdb分析[最推荐 企业级]** 使用开源工具 redis-rdb-tools或其他 离线解析rdb快照文件 
     - 完全不影响线程redis性能 可以导出所以大key详细csv报表
4. 治理策略
   - 拆分 比如一个10万的大hash 用取模算法拆分成100个小hash 每个hash存1000个
   - 压缩 对于纯文本大json 用java的snappy或者gzip进行压缩 读取再解压 
   - 避免全量操作：禁用hgetall smembers keys*  使用hscan sscan 分批次迭代读取
   - 异步安全删除 redis4.0以上 不用del删除大key 强制要求使用unlink 会把解除关联操作放在主线程 释放内存给后台线程异步执行 防止阻塞。



## 热key

在某个时间内 某个特定的key带来了海量的并发请求(高qps)  重大新闻热搜 秒杀爆款商品



1. 危害
   - 打爆单点  redis集群也害怕并发 因为经过hash槽计算 这个热key永远只会打向一台机器 几万qps会直接把一个节点的cpu和网卡打满 节点瘫痪
   - 引发雪崩 节点瘫痪或者热key突然失效 流量会穿透redis 把底层mysql打死 。典型的缓存击穿引发雪崩。
2. 排查热key
   - redis-cli --hotkeys 前提是内存淘汰策略是lfu
   - 客户端本地收集：侵入性强  在redis客户端(lettuce/redisson)进行拦截 用一个本地滑动窗口统计每个key访问次数 超过阈值 报警
   - 代理层发现 :最优雅 有redis代理网关（proxy。统计代码写在proxy 对业务代码完全透明
3. 解决方案

解决思路就是防线前置

- 引入jvm本地缓存 (最好用)
  - 机制:系统检测到某个热key 加载到应用服务器的本地内存
  - 效果：有100台tomcat服务器 热key流量进来 直接在tomcat本地返回 不会去打redis
  - 折中 需要设置一个极短的本地过期时间(1-3s) 接受这几秒内的数据短暂不一致 (最终一致性)
- 打散热key
  - 机制：不方便用本地缓存 可以复制多份热key 分散到集群的不同节点 热key带后缀
  - 效果： 客户端请求的时候 在代码里随机生成不同的后缀 把原本打向单个节点的qps 分散到不同的节点。
- 兜底 降级限流
  -  引入组件，当热点接口的qps超过系统极限 直接返回 系统繁忙，宁可让部分用户体验不好 也要保护数据库不挂



## 总结

大key 核心是拆分 压缩 异步删除 关键在于避免阻塞主线程

热key 核心是集群无法分摊单点qps 最佳方案是引入多级缓存拦截拦截流量 配合打散副本和 限流兜底保证高可用
