# redis cluster

主从复制和哨兵机制是为了解决单机故障问题

切片集群 是为了解决 单机内存和性能瓶颈 问题

情景：

- 当公司业务数据大到几百GB甚至上TB 一台服务器内存装不下。 
- 并发写操作极高 一台机器的cpu扛不住 
- 就必须用切片集群分担压力到其他物理机。

## hashSlot

固定哈希槽位位16384 

1. 路由算法

   - redis存入一个键值对的时候 redis会拿键名去做 CRC16算法运算 得出一个16bit的数字

   - 数字对16384取模 得出key对应的哈希槽编号

2. 节点分配算法 比如有三个物理机
   - 平均分配 16384/3 一台拿5461
   - 手动分配 比如两台老机器 一台新的 希望新机器多干活 可以给新的多分点位置

3. 这是一个逻辑概念 并没有实际的槽位 只是把一个键值对映射到到槽位 --> 节点 

## 概念

1. 一个物理机是一个承载节点的容器， 会有很多节点 ，可能是主node 也可能是从node 。所有物理机的node共同组成集群。

2. slot是分配给redis node 是一个在特定端口运行的redis进程，不是直接分配给物理机的。

3. cluster采用主从节点模式 并且不采用哨兵模式。也就是一个主节点对应一个/多个从节点。

4. 为了防止一台物理机断电导致数据丢失，主节点和从节点不会放在一台物理机上。所以物理机上的node是集群 但是没有对应的主从关系。

   - 假设你有 6 台物理机，标准的部署是：机器 1、2、3 分别部署主节点 A、B、C（它们三个平分 16384 个槽位）。机器 4、5、6 分别部署 A、B、C 的从节点。

   - 这 6 个节点共同组成了一个庞大的 **Redis Cluster（切片集群）**。

5. 当主节点失效的时候，采用的是所有主节点互相监督，选举出其从节点当选新的主节点。切片集群内部集成了类似哨兵功能。

   - 集群里的各个主节点之间，会通过 Gossip 协议不断地互相发送心跳包（PING/PONG）来监视对方。

   - 如果主节点 B 突然宕机了，主节点 A 和主节点 C 发现 B 没动静了，它们就会内部投票。确认 B 真的死透了之后，A 和 C 会直接指挥 B 的从节点你现在升级成新的主节点 B

   - 自带了故障发现和自动转移功能，根本不需要额外部署一套哨兵系统

## 优点

1. HA：横向拓展内存 自带哨兵机制。
2. 高性能：以前是所有客户端对一台机器读写 现在是几台 几十台同时处理请求 QPS成倍增加
3. 扩展性好 ：当需要提升的时候，直接加几台机器进集群，然后把旧机器的一部份哈希槽包括数据迁移到新机器 。不需要的 也可以下线结点 很方便



## 缺点

1. 部署和维护复杂 需要监控几十个节点的所有问题 运维难度大
2. 数据同步延迟 节点越多 网络通信越复杂 如果某个结点挂了 引发主从切换 或者进行槽位迁移 都会占用大量的网络带宽和磁盘IO 这个此前可能会出现短暂的读写延迟和 报错
3. 某些操作不能跨节点实现。比如一行指令同时涉及到node a & node b redis集群默认不支持跨多node进行事务操作或者多键操作 则会报错
