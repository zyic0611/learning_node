Q：Redis 中的 `ZSet`（有序集合）底层是如何实现的？为什么 Redis 在设计时，对于数据量较小的 `ZSet` 没有直接采用跳表（SkipList）？

A：redis中zset在数据量较小或者元素较短的时候，采用ziplist/新版本替换为listpack实现 因为其内存连续 不仅节省了大量指针带来的内存开销 还可以更好的利用cpu缓存命中率【局部性原理 当数据量达到阈值的时候 底层会切换为skiplist和dict【hash  跳表负责范围查询和排序 时间复杂度为LogN 而dict负责维护元素到分数的映射 保证了zscore这样的单点查询保证O1的时间复杂度

Q：在实际的高并发业务中，你是如何理解并解决“缓存穿透”和“缓存击穿”问题的？如果系统中引入了布隆过滤器（Bloom Filter）来防止缓存穿透，它在实际业务维护中有什么明显的局限性？

A：

1. 缓存击穿 就是比如一个热key 高频访问的redis key 过期了 则大量的请求打到mysql 造成mysql连接池一下子被打满 宕机了。 解决方法就是 加分布式锁。给向mysql的请求加锁，保证只有一个查询能到数据库，让其他请求线程休眠一小会，然后再去重新查询redis 此时应该第一个线程给mysql的查询请求结束，并且写回了redis，其他线程都查到了，就保护了mysql。并且返回的数据是最新的。 或者给数据加逻辑过期时间，当访问的redis数据逻辑过期了，就直接返回旧数据，并且开启一个线程去数据库查询新数据，这个线程也得上分布式锁，这种方式响应快，但是返回的数据是旧的。
2. 缓存穿透就是 向redis查询不存在的数据，数据不存在则会访问mysql mysql里也没有 就不会返回数据写入redis 如果不停的查询不存在的数据 则所有请求都到达了 mysql ，这就是缓存击穿。解决办法是缓存空对象，如果没查到就返回null 在redis里存储 但是大量的存储空对象 会造成内存浪费，则可以使用布隆过滤器。
3. 局限性就是不适合删除数据，因为里面的1标记可能是多个数据打上去的，删除对象的时候不能删除布隆过滤器里的标记。 只能在半夜的时候 重新同步一下数据库的内容来刷新布隆过滤器。 并且当数据量很大的时候，1的标记很多，访问不存在的对象 通过bloom过滤器的概率也很大。
4. 所以针对他的痛点 可以引入布谷鸟过滤器 或者在凌晨业务低峰期 异步线程全量重建布隆过滤器  对于误判 可以适当增加hash函数的个数 或者扩大位图的容量

Q：很多后端业务在处理资源互斥时会用到 Redis 做分布式锁。请描述一下你实现 Redis 分布式锁的核心思路（需要用到哪些命令）？如果业务代码执行时间过长，导致锁自动过期释放了，这种“锁超时”问题应该如何妥善处理？

A：

1. 分布式锁的核心思路： 核心依赖 set命令的 NX【not exists 和px【毫秒级过期时间   set lock_key value nx px 30000 来保证加锁和设置过期时间的原子性 释放锁的时候 不能直接del 必须用lua脚本判断当前锁的uavlue 比如UUID+线程ID 是否和预期值一致 一致才能删除 防止误删了其他线程的锁。
2. 引入redisson 使用看门狗机制 很好的解决锁超时的问题 设置锁的时候 设置过期时间为-1即可。原理是在加锁成功后 后台会启动一个定时任务 默认每10秒执行一次 去检查持有锁的线程是否还在运行 如果还在 就把过期时间续期位30s 既防止死锁 又完美解决业务执行过长导致的锁提前释放的问题



Q：如果 Redis 主从架构下，主节点挂了导致分布式锁丢失（脑裂问题），应该怎么解决 【分布式锁安全性问题

A：

1. 丢锁的原因 主从复制是默认是异步的。A线程在master节点成功加了锁 但是master还没来得及同步给slave master突然宕机了 此时哨兵机制 slave升级为新的master。接着 线程b请求同一把锁 master发现自己没有这把锁 则允许b加锁  导致了a 和 b同时持有一把锁 分布式锁的互斥性被打破了
2. redis官方使用红锁 redlock  抛弃主从架构 使用多个独立的redis master节点 通常5个  当客户端需要加锁的时候  只有当半数以上master加锁成功 并且整个总 获取锁的时间没有超过锁的有效时间 才会认为分布式锁获取成功 这样即使有1-2两个节点宕机 只要多数节点存活 锁依然可靠
3. 实际业务开发中 很少使用 redlock
   - 维护成本太高 需要5个独立master节点 
   - redlock强以来服务器失踪 发生时钟跳跃或者长时间JVM GC暂停 仍然极小概率导致锁失效。
4. 真实架构：
   - AP场景 追求极端高并发 允许小概率不一致   电商秒杀库存比如 会使用redis主从架构+redisson 为了兜底很小概率的丢锁情况 在数据库层面或者业务代码层面做好幂等性校验【比如数据库唯一索引 乐观锁 用很低的成本换取高性能
   - CP场景 绝对一致性 不能丢锁 比如金融打款 跨境交易 不适用redis  **果断切回到 ZooKeeper 或者 etcd**。它们底层基于 Paxos/Raft 共识算法，只要写入成功，就一定同步到了多数节点，天然保证了强一致性，从根本上杜绝了脑裂丢锁的问题。

Q：CAP理论是什么

A： 是分布式架构的最基本的铁律 一个分布式系统 只能最多满足两个

- C (consistency) 一致性   读取最新的数据
- A availability 可用性。  对用户的请求给出相应 不报错 不超时 数据不是最新的
- P  （partition tolerance）分区容错性 节点直接因为网络抖动断开连接 形成网络分区 系统仍然要可以运行

实际分布式系统中 P是必须保证的 网络抖动是常态。

- AP架构 高可用 牺牲强一致。比如评论区 redis 网络断了 仍然能刷出旧评论 保证A 网络恢复再去同步
- CP架构 强一致 银行转账 zookeeper 网络出现问题 系统宁愿报错 也不看到错误数据

Q：大家都知道 Redis 内存满了会触发淘汰策略，比如 LRU（最近最少使用）。Redis 内部实现的 LRU，是教科书里那种标准的、基于双向链表+哈希表的 LRU 吗？** 如果不是，它是怎么实现的？Redis 为什么要放弃标准的 LRU 算法而采用这种特殊设计？

A：

LRU是 least resently used 是最近最少使用 强调时间  LFU是 least frequently used 强调频率/次数

不是 是一种类LRU算法  在对象头中维护一个24位的字段 记录数据最后一次访问的时间戳。当内存不足触发淘汰时，redis会随机采样N个节点，（默认是5个）出来，把他们放在一个按空闲时间排序的驱逐池，【Eviction pool，默认大大小为16，淘汰掉池子里最久没被访问的key，这种做法在很低的内存占用小，达到了和标准lru非常接近的效果。因为标准的lru占用内存空间大，双向链表 一个节点又2个指针 。乳沟是lfu的话 24bit位分出8bit存逻辑访问次数。

Q：关于 RDB 持久化，我们知道 `bgsave` 命令会在后台异步生成快照。面试官的问题是：**执行 `bgsave` 时，底层调用的 `fork()` 机制和写时复制（Copy-On-Write, COW）技术是如何工作的？** 

A：

1. fork（）会开启一个**子进程** 异步生成快照。redis之所以能保证单线程执行命令不阻塞 就是用到了 操作系统的多进程机制去做持久化。它不会立刻复制主进程的10GB内存 而只是复制主进程的页表【Page Table 此时 主进程和子进程共享同一快物理内存
2. cow技术就是当 生成快照的时候  没有写请求，就没事。如果主进程收到了写请求， 操作系统会把要修改的页单独复制一份出来给主进程修改 子进程依旧读取原本的数据进行快照。

Q：追问一个极端场景：* 假设你的 Redis 实例当前占用了 10GB 的物理内存，而此时服务器只剩下 2GB 的空闲可用内存了。这个时候执行 `bgsave`，会导致内存溢出（OOM）崩溃吗？为什么？

A：不会立刻崩溃  frok瞬间并不需要额外内存 。取决于bgsave执行期间的写并发请求，如果在快照期间，主线程收到了大量的写请求，导致被修改的内存也 超过了剩下的2GB 也就是产生了大于2GB的COW内存拷贝 则会超过物理内存限制 导致OOM 导致redis崩溃 反之则安全。

Q：在真实的线上环境中，**“Big Key（大 Key）”** 和 **“Hot Key（热 Key）”** 是导致 Redis 集群雪崩的两大杀手。 请分别解释一下什么是 Big Key 和 Hot Key，它们各自会导致什么严重的线上问题？如果你在负责这个后端项目，你会用什么方案来**发现**并**解决**这两个问题？

A：

1. 大key主要是指 value的体积过大 比如一个string超过10MB 或者一个hash有几百万个元素。 问题是在单线程模型下，导致读或者删除的的时候严重阻塞主线程 。
2. 排查方法是：使用自带的redis-cli-bigkeys  进行扫描  抓去rdb备份文件 使用开源工具rdbtools进行离线分析 可以不影响线上性能
3. 解决方法是：业务上对大key进行拆分 拆成多个小hash 最重要的是不适用del 使用redis4.0引入的unlink进行异步惰性删除 lazy free 交给后台线程释放内存
4. 热key就是redis里瞬间被极其高频的访问  会导致 单台reids节点 网卡被打满 cpu飙升 热key一过期 流量全部涌入db 导致db宕机。 
5. 排查方法是：开启lfu后 可以用 redis -cli --hotkeys扫描 在客户端/proxy代理成进行访问拦截统计
6. 解决方法可以采用多级缓存，比如JVM本地缓存caffeine 或者在mysql做限流兜底 当流量过大的时候 就牺牲用户体验 拒绝访问mysql 或者分散热点key 复制几份 把流量打散到集群的不同节点 分散单节点压力。 



Q：Redis 官方提供了 **Sentinel（哨兵）** 和 **Redis Cluster（切片集群）** 两种高可用方案。请问这两种方案的核心定位有什么不同？在什么业务痛点下，我们会放弃使用哨兵模式，强制要求将架构升级为 Redis Cluster？

A：

哨兵机制是为了解决单机故障的情况，通过哨兵机制实现故障转移。

切片集群主要是为了解决单机性能瓶颈的问题，通过拓展节点，物理机。

在单机结点上，内存往往是硬伤。由于redis是单线程，如果单节点内存分配过大，比如超过10GB 在执行bgsave 甚至日常的持久化的时候 都会引发严重的性能抖动。所以当业务的数据总量或者并发QPS超过单台物理机的极限的时候 就会强制切片上cluster。

<u>Q：【数据路由与分片底层】 在 Redis Cluster 中，数据是如何进行分片存储的？ *追问底层原理：* 业界在做分布式缓存分片时，很多都会采用“一致性哈希算法（Consistent Hashing）”来尽量减少节点增删时的数据迁移。**但为什么 Redis Cluster 偏偏放弃了一致性哈希，而是采用了 `16384` 个哈希槽（Hash Slots）的机制？**</u>

A：

防止一致性哈希 核心是为了 **数据迁移和 集群扩缩容** 的极简性* 一致性哈希在删除或增加节点的时候 会导致相邻的节点数据发送不可控的链式迁移，运维成本很高。 redis cluster 采用16384个固定的哈希槽 将数据与槽位绑定 槽与节点绑定。 当集群需要扩容的时候 只需要明确的从老节点中剥离一部分槽定向迁移给新节点 这种解藕的设计 让数据的重新分配变得极其精确并且易于操作。

Q：Redis Cluster 并没有像哨兵模式那样有一个“第三方管理者”来监控节点存活，所有的节点都是平级的。 请问集群中的节点之间是如何互相发现并同步状态的？（提示：Gossip 协议）。如果其中一个 Master 节点物理机突然断电宕机了，整个集群是如何在内部通过通信发现它挂了（`pfail` 到 `fail` 的转变），并最终选举出新的 Master 恢复服务的？

A： 

redis cluster是一个去中心化的架构 节点之间通过Gossip协议 主要是PING/PONG消息 来不断交换彼此的状态。

1. 主观下线 PFAIL【possible fail ：master节点每秒会随机ping几个节点 如果a发现b在设定的时间没有恢复pong a就会在本地状态里把b标记为pfail
2. 客观下线 FAIL a通过gossip协议把b pfail的消息广播给集群中其他master 当半数的master都报告说收到了b的pfail 则b被认为fail
3. 投票： b被fail以后 b的所有slave会发起选举 通过其他所有存活master的投票 拿到半数以上选票的master会成为新的master

Q：一致性哈希介绍一下：

A： **最终点的问题 就是影响邻居**

原理：

- 是一个Hash Ring 比如由2的32次方个节点组成

- 根据节点的ip地址 进行哈希运算 映射到圆环的几个位置上

- 当有key存入 对key进行hash 落到圆环某个点上

- 顺时针寻找 遇到的第一个节点 就是key存储的归属地。

删除节点：

- 如果A突然宕机/被删除 按照顺时针查找规矩 原本属于A的所有数据 都会顺时针找到B 也就是存在B区 
- 看起来比 Hash%n 好多了 传统方式下 n变了 所有数据都要迁移 一致性哈希只影响了宕机结点相邻的下一个节点 但是会出现雪崩【级联故障 【不可控的链式迁移 
  - 如果平时每个结点的负载时70% A崩溃后 A的全给B B就140%
  - B也宕机了 则顺时针都给C C也宕机

增加节点：

- 如果A B C 负载都很大 则在A B中间插入一个节点X
- 按照规则 X 只会截胡本来从 A要流向B的一部分数据
- 结果就是X 只帮B分担了压力 
- 无法精确的控制新节点均匀的去分摊集群中 所有老节点的压力

对比哈希槽：

- 槽与节点解耦 槽不是顺时针自动分配 是人为显式分配 
- 扩容 加一个节点D 管理园就命令 A给1000个槽给D B C同理
- 整个数据迁移是确定的 点对点的 完全受控的 



Q：大家都知道 Redis 的绝大部分操作是单线程的。面试官问：**既然是单线程，为什么它能扛住十万甚至百万级的并发？底层是如何利用操作系统的 IO 多路复用（比如 Linux 的 `epoll`）机制的？** 

是因为他是纯内存操作 速度快。单线程模型避免了多线程情况下频繁的上下文操作 免去了加锁等复杂操作。以及极其高效的底层数据结构，还有IO多路复用 是哟哦那个操作系统底层的epoll操作 监听所有服务器，也就是所有socket文件描述符【FD 只有当有数据可读或者可写的时候 才会通知主线程 主线程只被动响应 不主动询问 实现了单线程高效的并发管理多个线程。避免了主线程无效的轮询。

Q：*追问极限边界：* 既然单线程这么完美，**为什么 Redis 6.0 之后又引入了“多线程”？** 这个多线程到底是用在处理什么任务上的？

因为在万兆网卡时代，网络IO读写和协议解析成了瓶颈，多线程主要是reactor 一主多从框架  主线程只负责连接和核心业务处理。当发现需要解析网络协议和读取网络数据的时候 交给子线程去执行。大幅度提升了吞吐量，也完美避免了多线程加锁带来的复杂性和性能损耗。

Q：假设你正在小红书或抖音负责用户行为分析系统。

- **场景 A：** 统计一整年 365 天里，某个用户的**连续打卡/签到情况**，要求极其节省内存。你会用什么数据结构？怎么算连续签到天数？到 
- **场景 B：** 统计今天 App 的 **DAU（日活跃用户数）**，日活上亿，要求去重，并且对内存有着极其变态的限制（比如只给你十几 MB 内存，绝对不能用 Set），即使有极小概率的误差（1% 以内）也可以接受。你会用哪种数据结构？它底层大概的运作逻辑是什么？

Q： 之前我们聊过 RDB 可能会丢失最后一次快照后的数据。为了保证绝对安全，我们会开启 AOF（Append Only File）日志。

- **痛点：** AOF 文件会越写越大，Redis 是如何通过 **AOF 重写（AOF Rewrite）** 机制给文件“瘦身”的？如果在重写期间（依然是子进程在做），主进程又有新的写请求进来了，这部分**新数据会丢失吗？Redis 是怎么解决这个差异的？**
- **终极方案：** 纯 RDB 恢复快但丢数据，纯 AOF 安全但恢复极慢。**Redis 4.0 引入的“混合持久化”是如何将这两者的优点完美结合的？**

A：

1. AOF重写并不是扫描旧的aof文件 去优化 而是子进程直接扫描当前内存里的数据 生成新的写入数据。 为了防止子线程在重写期间 主线程收到新的写请求 redis设计了两个缓冲区
   - 主线程会把新的请求 放入普通的aof 缓冲区 保证了旧aof文件的安全性
   - 还会把新写请求放入专门的 aof重写缓冲区  
   - 等子进程把内存快照转成新的aof文件后 主进程会吧aof重写缓冲区里的增量数据 追加到新aof文件 最后原子替换旧aof文件 做到数据0丢失
2. 混合持久化
   - 当开启混合出久化后 aof重写的工作方式改变
   - 子进程开始aof重写的时候 不再把内存变成一行一行的aof命令 而是直接以rdb快照【二进制格式的方式 写入新的aof文件里。
   - 在子进程写入rdb快照的时候 主线程 的新写请求 依旧同时以aof格式写入两个缓冲区
   - 待子线程写好rdb文件后 把增量数据 以aof格式 追加到文件末尾
   - 重启回复的时候 先光速加载前半段rdb 后重返后半段少量的aof 
   - 实现了重启快 不丢数据

Q：Redis 支持事务（`MULTI`/`EXEC`）吗？它的事务和 MySQL 的事务（ACID）有什么本质区别？** *致命追问：* 如果在事务执行期间，某条命令由于语法错误或者类型不匹配报错了，**Redis 会像 MySQL 那样触发回滚（Rollback）吗？** 在大厂高并发业务中，如果我们需要保证多个 Redis 操作的绝对原子性（比如前面提到的释放分布式锁），我们真正会采用的标准方案是什么？

A：支持使用multi exec 开启事务 提交事务。但是不支持事务回滚。如果事务执行期间 有编译器间错误 则直接报错 事务结束。如果已经执行了几条代码，再出现错误，也不支持回滚，执行完的仍旧成立。 我们会采取lua脚本，保证几条redis操作的原子性。redis只能保证单条指令吧的原子性。

Q：假设线上有一个极其耗时的批量查询接口，你需要一次性从 Redis 读取或写入 10000 个毫无规律的 Key。 如果你用 `for` 循环调 10000 次普通的 `GET/SET`，**网络往返的时间开销（RTT**）会把系统直接拖死。 *提问：* 除了原生的 `MGET/MSET` 命令之外，**Redis 还提供了一个什么重量级技术来解决这种“海量碎片化网络 I/O”问题？它和原生 `MGET` 有什么本质区别？**

A：

为了解决海量命令的网络往返耗时 会使用pipeline【流水线技术。

本质上是客户端层面的一种优化  会把海量命令打包成几个大的TCP数据包一次性发给redis redis执行完 再把10000个结果一次性打包返回 大大降低网络IO开销。

与原生MGET的区别

1. MGET是redis原生支持的一个原子命令 只支持String类型的读
2. pipeline只是网络打包 不是原子的。里面可以随机搭配各种类型的命令 在redis服务器看来 依旧是一个个独立的命令 中间可能会穿插执行其他客户端请求。

与lua脚本区别

1. lua脚本主要是为了解决 多条redis命令的原子性问题 虽然支持一次性发给redis 但是都是毫无规律的key 会导致redis单线程长时间阻塞 其他客户端请求全部超时
2. 所以多条无规律 海量命令 用 pipeline 少量原子性  用pipeline

Q：很多业务初期为了省事，不想引入 Kafka 或 RabbitMQ，就会用 Redis 的 `List`（利用 `LPUSH` 和 `BRPOP`）或者 `Pub/Sub`（发布/订阅）来客串消息队列。 *提问：* **这两种老旧的方案在做 MQ 时，分别有什么致命缺陷？** （提示：跟消息丢失和消费组有关）。 *架构进阶：* 为了彻底解决这些缺陷，**Redis 5.0 专门引入了一个什么全新的数据结构？** 它在底层设计上几乎“抄”了 Kafka 的作业，成为了轻量级 MQ 的完美平替。

A：

“业务初期确实可以用 List 或 Pub/Sub 客串 MQ，但它们都有致命缺陷：

- **List（`LPUSH` + `BRPOP`）：** 最大的缺陷是**不支持广播模式**（一个消息被弹出后就没了，不能被多个微服务同时消费），而且**没有 ACK 机制**。如果消费者刚拿到消息，还没处理完就崩溃了，这条消息就永久丢失了。
- **Pub/Sub（发布订阅）：** 最大的缺陷是**极其不靠谱的‘发后即忘’（Fire and Forget）**。它不持久化消息，如果消费者因为网络抖动短暂离线了，这期间发布的所有消息对它来说都直接丢失了。

**终极架构进阶：** 为了彻底解决这些痛点，Redis 5.0 专门引入了一个重量级数据结构 —— **Stream**。 它在底层设计上深度借鉴了 Kafka：

1. 支持**消息持久化**（可以用 `XADD` 追加消息，宕机不丢）。
2. 支持**Consumer Group（消费组）**，同一个组内的消费者可以竞争消费（负载均衡），不同组的消费者可以独立消费同一份数据（广播模式）。
3. 完美支持 **ACK 机制**（使用 `XACK`）。如果消费者处理失败，消息会留在 Pending 列表里，随时可以重新认领处理，彻底杜绝了消息丢失。”
