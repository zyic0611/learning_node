# SDS

c语言String的问题

1. \0结尾
2. 获取长度遍历寻找\0 On redis里执行strlen 要遍历全表 效率低
3. 容易内存溢出或者越界 c语言不记录数据分配多大内存
4. 二进制不安全 存图片 压缩流 可能一堆\0 读取不了



## 结构

**`len` (长度)：** 记录当前字符串**实际**使用了多少字节。

**`alloc` (容量)：** 记录当前 SDS 内部**总共分配**了多少字节的空间。

**`flags` (标志)：** 极其抠门的设计！用来标记当前 SDS 是什么类型（比如 `sdshdr5`、`sdshdr8`、`sdshdr32`）。意思是：如果字符串很短，就用 8 位的数字来记长度；如果很长，再用 32 位的数字。能省一个字节是一个字节！

**`buf[]` (字节数组)：** 真正用来存数据的字节数组。**（注意：SDS 依然保留了在数组最后加一个 `\0` 的习惯，这是为了能兼容部分 C 语言的底层基础函数库，但这不影响 SDS 本身的逻辑）。**

## 优势

1. O1 读取len
2. 彻底杜绝内存溢出 给sds拼接新的字符串 sds的api会检查alloc-len 剩余空间足够与否 不够会自动扩容
3. 二进制安全 可以读取各种二进制流

## 内存管理的优化

1. 空间预分配 修改时扩容

   - 修改后len<1MB 则直接分配两倍空间 下次追加就不用申请内存

   - \>=1MB 则多分配1MB

2. 惰性空间释放 缩短时保留:

   - 当缩短SDS的字符串 比如删除了一段 redis不会立刻归还内存给OS 预防后面还要追加数据  省去了频繁释放和申请内存的开销
