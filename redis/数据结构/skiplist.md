# skiplist

是一个单向链表链表内的工作机制，理解为加上了多层索引机制。  redis做了特殊优化 变成了双向链表

理解为抽出部分结点 建立额外的连接。作为第1层，第二层，第三层。



## 寻找数字

1 .要寻找一个数字的时候，总是从最顶层的索引开始搜索。

2. 在顶层跨越大量结点，如果发现下一个节点>目标结点，说明目标在当前节点和下一个节点内。

3. 此时，指针降级到下一层寻找。

4. 一路降级到第0层，精准锁定目标，原本需要遍历n次，现在可能只需要跳跃几次，比对几次，时间复杂度降到logn，与二份查找同一级别。

## 插入结点

- 新节点插入 默认在第0层
- 抛硬币（概率为1/4  
- 如果为正面 就加一层索引
- 反面 结束抛
- 最高限制为32层 redis5.0后修改为64

## 为什么不用红黑树或者b+

1. 范围查找效率最高！！
   - zset底层使用跳表 zset最核心的功能是类似于ZRANGEBYSCORE这样的范围查找 
   - 使用红黑树 找到下限很快 但是要继续获取后续元素 需要复杂的中序遍历
   - 使用跳表 找到下限后 由于第0层本身就是完整的双向链表，顺序遍历取值即可。
2. 实现简单
   - 红黑树插入删除 维持树的平衡 复杂的左旋右旋 代码实现难度高
   - 跳表的插入只要修改相邻节点指针 代码少 由于随机层数 不需要严格维护整棵树的平衡
3. 内存占用更合理
   - 跳表通过调整晋升概率来控制索引节点的数量 相对于红黑树必须在每个节点都存储左右节点指针和颜色标记位 跳表在指针内存占用上更省
4. 为什么无所谓索引高度
   - mysql使用 B+树 是放在磁盘里 需要尽可能减少io次数 所以要尽可能减少树高
   - redis在内存里 读取速度快 可以不在乎层高



## 特殊设计

1. 自带后退指针 标准跳表是单向的 redis为了能支持反向范围遍历 比如寻找小于90的 在最底层/第0层加了一个后退指针 形成双向链表 
2. 分数和对象的双重排序 :节点不仅按照 `score`（分数）从小到大排序；如果 `score` 相同，还会按照成员对象（`member`）的字典序进行排序，保证了数据的绝对唯一和有序。
