# 索引调优

## 索引失效

1. 给字段 加上了 函数或者数学计算
   - **数学计算：** `SELECT * FROM user WHERE age + 1 = 19;`
   - **套用函数：** `SELECT * FROM user WHERE LEFT(name, 1) = '张';` (假设 name 有索引)
   - **结论：绝对不要在where等号的左边【索引列 做任何的运算或者函数操作 必须把计算移动到等号右边 比如改成 `WHERE age = 19 - 1`**
2. 隐式类型转换
   - phone的类型为varchar 并且建立了索引
   - SELECT * FROM user WHERE phone = 13812345678
   - mysql发现两边数据类型不一致 在底层做了一个自动类型转换【cast 。
   - SELECT * FROM user WHERE CAST(phone AS SIGNED) = 13812345678;
   - 此时就变成第一种情况 对索引进行运算
   - **结论：字符串类型的字段，查询的时候 必须加单引号 避免隐式类型转换  改成 `WHERE phone = '13812345678'`**

3. like模糊查询 -最左前缀法则
   - **SQL A：** `SELECT * FROM user WHERE name LIKE '张%';`
   - **SQL B：** `SELECT * FROM user WHERE name LIKE '%三';`
   - B+树不仅对联合索引讲究从左到右 对单个字符串的字段的内部排序 也讲究从左到右 
   - 所以第一个可以走索引 第二个就是全盘扫描。

## EXPLAIN

### 索引失效总结：

- 在列上做计算/用函数失效。

- 隐式类型转换失效。

- 左边带 `%` 的 LIKE 失效。

- 联合索引不满足最左前缀失效。

有些复杂的sql是看不出来索引是否失效 就需要explain 执行计划。 mysql就不会去执行这条sql 而是返回一个报告。

1. type 查询类型 
2. possible_keys mysql优化器可能用到的树
3. key 世纪用到的索引 为null 说明全盘扫描

### type：

- all 全盘扫描 
- index 全索引扫描 没扫全表 但是查全部索引树 通常查询字段都在索引树上 就直接全部扫索引树了 不需要回表
- range 范围查找 between > < in()
- ref 索引的等值查找 可能不是唯一的 会查出多条数据 
- eq_ref  连接查询 在使用join的时候 主表在从表中只会匹配到一条记录 最多 通常是 从表使用了主键/唯一索引 作为关联条件：SELECT * FROM a JOIN b ON a.id = b.a_id;
- const 极速匹配 通过主键或者唯一索引 由于是唯一确定的 扫描到一条就停止
- system 表里只有一行数据 const的特例

性能从低到高排序

### using 

- using index: 覆盖索引 没有回表
- using index condition 索引下推 回表了 但是在二级索引上过滤一批
- using where 在mysql 的 server做where过滤  不一定没用索引 有可能用了索引  但是可能只过滤出来了部分条件 还需要丢给server层走where条件再过滤。
- using filesort ：mysql无法利用索引排序 只能把数据读取到内存中 在应用层进行一次排序 非常耗CPU和内存
  - SELECT * FROM user WHERE age = 20 ORDER BY create_time
  - 只有age是单列索引 扫出来数据后 create_time是乱序的
  - 建立联合索引 就天然有序了
- using temporary 使用临时表 为了完成查询 得建立一张内部临时表来存放中间结果 用完之后要删除
  - group by distinct 多表join 再排序都会引发
  - 让group by 的列走索引 比如分成 group by 分成3个 就需要建立三个临时表 但是如果走索引 顺序读取就好了 天然有序

总结：如果type是all 或者index 或者 extra是 filesort/temporary 检查where 和order by 建立/调整联合索引 或者用最左前缀消除

## 要不要建立索引

### 看区分度建立索引

- 性别 状态...区分度低的字段 不要去建立索引 b+树无法快速缩小范围 最后的回表成本比全盘扫描还大 因为走索引树相当于是疯狂的去随机I/O 而全盘扫描时顺序I/O 性能更高。
- 身份证号 手机号 流水号 高度唯一 区分度高 适合用B+树

### 索引不是越多越好

- 空间代价：每一颗索引树 都是16KB的一个结点 堆砌起来的一棵树 占内存
- 时间代价：索引树越多 insert delete update数据 维护的树结构越麻烦 【页分裂问题 5颗索引树同时分裂 写性能很差



### Q：怎么决定建立哪些索引 索引是不是建的越多越好呢？

A：不是 索引本质上是利用空间和**写性能**去换读性能。 建立索引的时候 要评估字段的区分度。区分度低，反而不如全盘扫描。 并且 要尽量利用 联合索引和 最左前缀法则 复用树结构 争取触发索引覆盖 尽量减少树的数量 平衡写性能。
