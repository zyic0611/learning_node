# 数据库的并发

## 三大并发读写问题

1. 脏读 【读未提交
   - 读数据的时候读到了别的事务修改的数据 但是还没这个事务还没提交 数据库根本不存在 
2. 不可重复读
   - 在一个事务中 查询了两次数据 查询的是同一行 两次数据不一致
   - 也就是在两次查询的过程中 数据被人修改 提交了【commit 
   - 侧重于一行数据被人update了/delete 查不到或者查的不一样 是不可重复度
3. 幻读 
   - 在一次事务的两次查询中 查询出来的数据 条数不同 
   - 侧重于数据被人insert  查的更多了第二次 是幻觉

## 隔离级别

1. 读为提交 RU ：什么都不防
2. 读已提交 RC ：解决了脏读 只允许读取别人提交的数据
3. 可重复读 RR：解决了脏读+不可重复读 是Mysql InnoDB的默认隔离级别
4. 串行化：无并发可言 性能低 不用



## MVCC 并发读问题

主要是做到了 不加锁 还能读到旧数据。

InnoDB默认隔离级别是RR  意思是 医生在同一个事务里，哪怕病人已经把订单 UPDATE 成了“已取消”，医生第二次查，**依然必须看到它是“已预约”**。

这就有个大问题了：**数据库里真实的数据已经被改成了“已取消”，医生凭什么还能查出“已预约”？**

解决方法：

1.加锁 锁住这几行数据 不允许修改 但是并发性能很低 

2.MVCC 多版本并发控制 读写不冲突 

### Undo Log 【时光机 记录数据的历史

当一条数据的信息进行了修改之后 旧数据并不是直接删除 而是在底层建立一个版本链【链表

1. 最新版本 当前数据库里的真实数据
2. 上一个版本 旧数据 存在undo log里 【不会存数据库的最新数据在undolog里
3. 它们之间用一个隐藏的指针连接起来

### Read View 照相机 决定能看到哪个版本

1. 开启事务的时候 mysql会 read view 拍一下快照。这个快照的内容，就是用户此时的版本号！

2. 如果在事务中，用户去查询数据发现和自己快照中的版本不同，就知道是在事务开启后别人修改了数据，则用户还是以快照为主。

3. 由于最新版本对此时的用户不可见，就顺着undolog寻找旧版本，以旧版本的数据为准。

4. 则用户读取了旧数据，并且整个过程没有加锁。读写完美并发。

### RC和RR的readview区别 快照读

MVCC在普通查询不加锁的操作 利用MVCC查询历史版本的专业术语叫做快照读。

**RC（读已提交）** 和 **RR（可重复读）** 底层都是用 MVCC 来实现的。它们的区别仅仅在于**“拍快照（生成 Read View）的时机”**不同。

假设现在有一个事务 A（医生查询），和一个事务 B（病人修改）。

1. 医生开启事务，执行了**第一次查询**（看到订单是“已预约”）。
2. 病人开启事务，把订单改成了“已取消”，并且**提交了**。
3. 医生在同一个事务里，执行了**第二次查询**。

**问题：** 如果你是 MySQL 的架构师，你要怎么设计“拍快照”的时机，才能让：

- **在 RC 级别下：** 医生第二次查，能看到“已取消”？（发生不可重复读）
- **在 RR 级别下：** 医生第二次查，依然看到“已预约”？（实现可重复读）

由于用户以readview内容为准，所以

- 在RR情况下，医生只有事务的第一次select时候readview 一次，事务内的数据以readview为准。
- 在RC情况下，每次执行select旧重新readview一次。

### readview内容

readview的内容其实只是一串事务ID 【TRX-ID

活跃事务ID列表

- 也就是数据库里正在执行 还没提交的所有人的事务ID。
- 判断版本对不对 就拿要查询的数据的事务ID和id列表对比
- 如果在 说明是别人没提交的半成品 不可见
- 如果事务id比列表里最大的id还大 说明是新开启的事物 未来的东西 不可见
- 如果比列表里最小的id还要小 说明是历史确定的 可见 



## MVCC 并发写问题

要写数据的时候 如果出现幻读现象 也就是本来要写 10条数据 由于 update/delete 数据多了或少了 此时没办法通过mvcc解决 那么只能上锁。

举例：

```
UPDATE orders SET remark = '高危患者请重点关注' WHERE age > 60;
```

在这个医生执行更新的瞬间，另外一个 65 岁的老大爷新挂了个号（INSERT）。

如果靠 MVCC 的照片，医生是看不见这个新大爷的。但 UPDATE 是要去改磁盘里最新数据的呀！如果医生改完了，结果这个新大爷因为没被打上“高危”标签而出事了，这就是极其严重的**幻读（凭空多出一条没被处理的数据）**。

mysql的底线就是死守在RR级别 不能退让到串行化，所以此时就需要上（）锁了。

### 间隙锁(gap lock)+行锁(record lock)=临键锁(next key lock)

1. 假设age二级索引B+树 只有 50 58 68 75 4个数据 由于叶子结点是顺序连接的 所以数据间形成了很多间隙【GAP
2. 医生执行 age>60的update操作 mysql遍历索引树的时候 不仅会给68 70 这两个>60的数据上行锁
3. 还会关于>60的间隙上锁 58-68 68-75 75-inf 全部上锁 
4. 当insert的时候 比如65的时候 会落在58-68的间隙里 但是由于间隙锁 则插入的线程会被阻塞起来
5. 必须等医生事务commit 释放锁 才可以



## mysql在RR级别下解决幻读 总结

如果是**快照读（普通的 SELECT）**，它是通过 **MVCC（版本链 + Read View）** 机制，让事务只能看到自己开启瞬间的历史数据，巧妙避开了幻读，并且完全不加锁，保证了极高的读并发。

如果是**当前读（UPDATE / DELETE / INSERT / SELECT ... FOR UPDATE）**，MVCC 就没用了。MySQL 会极其强硬地祭出 **临键锁 (Next-Key Lock)**。它不仅锁住满足条件的真实记录（行锁），还会把记录周围的‘空隙’也死死锁住（间隙锁），让其他事务根本无法在这个范围内插入新数据，从而在物理层面强行防住了幻读！”

其实只是最大程度避免幻读 没有完全解决幻读。

当快照读和当前读交叉使用的极端情况下，会产生幻读。

### 举例 

事务 A 执行快照读（SELECT）：发现没有 ID = 5 的数据。（此时生成了 Read View）

事务 B 执行插入：`INSERT INTO table (id) VALUES (5);` 并且提交。

事务 A 脑抽了，执行了一次当前读更新：`UPDATE table SET name = '测试' WHERE id = 5;`（请注意，虽然 A 的 Read View 看不到 ID=5，但 UPDATE 是当前读，去磁盘直接找最新数据，所以 A 成功更新了事务 B 插入的这行数据！）

**关键来了：** 因为事务 A 更新了这行数据，这行数据的 `trx_id` 就变成了事务 A 自己的 ID。

事务 A 再次执行快照读（SELECT）：**它惊恐地发现，ID=5 的数据出现了！幻读发生了！**（因为自己修改的数据，对自己总是可见的）。

所以也就是当前读可以跳过快照。
