# redolog&binlog

## update语句的执行

在mysql中，每次更新一个数据，就是去更新B+树的一个节点，也就是一页16KB的内存。找到这一页内存，拿出来修改后，再返回硬盘。

如果1000个更新操作，则要进行1000次该流程。而由于内存不是连续的，是在硬盘中随机放置，所以这种操作叫做随机I/O 效率极低，如果进行1000个这种操作，数据库会卡死。

## Buffer Pool 内存缓冲池

1. mysql启动时候直接要了一大块内存
2. update数据时 mysql先把这一页数据从硬盘拿到内存
3. 在内存里修改 内存里修改速度很快 此时硬盘数据是旧的  内存里数据是新的 被称为脏页
4. 此时直接告诉用户 更新成功
5. 等大半夜没人使用系统的时候 才会把脏页慢慢刷回硬盘 提高了并发写的性能
6. 但是如果断电 则内存数据丢失 违背了 四大特性的D 持久性

## Redo Log 重做日志与 WAL机制

WAL机制 write-ahead logging 预写式日志

1. 当内存里修改数据的时候 也就是脏页诞生的时候 此时就会写重做日志： 在向用户返回success之前 不会与硬盘同步数据 而是在硬盘的Redo Log日志文件里 追加一句话：

2. 在多少页的第几个偏移位置 修改了什么

3. 只要这句话成功写进RedoLog 就可以返回给用户Success

### RedoLog写硬盘为什么不慢--随机I/O与顺序I/O

- 刷脏页是在硬盘上到处找pos 是随机写 效率很低
- 写Redo Log 日志文件是在硬盘里的连续空间 每次写 只要向文件末尾追加 这种操作时顺序写 
- 在机械硬盘上 顺序写的速度是随机写的几百倍 速度几乎接近内存

mysql依靠wal机制来保证事务的持久性 当事务提交后 innodb不会立刻将修改后的脏页同步到磁盘的B+树中 因为随机I/O太慢 而是以顺序io的形式把修改操作追加到redolog中 。只要这个操作完成，事务就算成功，就可以返回成功。就算发生断电，mysql重启也可以通过读取redolog的日子，把来不及写会磁盘的脏页重做出来。

redolog内容是强调对页面（内存）物理位置的描述。

## BinLog 归档日志

是用来记录数据库每一次修改操作 是记录sql语句。所以记录的是逻辑日志。

与redolog的区别

### redolog 物理日志

1. 归属 ：InnoDB存储引擎的保命符
2. 内容：记录的对硬盘上真实物理页的修改
3. 只是用来防止断电的 所以大小是固定的 写到末尾了 会回到开头覆盖写

### binlog 逻辑日志

1. 归属：server层的 用什么引擎都有这个日志
2. 记录的是逻辑操作指令 也就是sql语句
3. 追加式 不覆盖 满了就加文件数

### 配合

1. 只有redolog 没有binlog：

   - 生产中 订单很多 为了保证高可用 数据库不止一台 必须是一主多从 一般是主用来写 从用来读

   - 覆盖过的redolog丢给slave 或者给dba 恢复3天前删除的数据 做不到 因为历史数据被抹除了 而且slave的物理布局可能与master 不一样 看不懂物理日志

   - binlog是追加写的 

     - 主从复制(集群同步) 主库把逻辑指令给从库 从库照着指令重新执行一遍 完成同步
     - 数据归档 只要拿出三个月前的**全量备份**，理解为当时的状态。再加上这三个月以来所有的 Binlog，从头到尾重放一遍，就能把数据库恢复。
       - redolog是做不到数据归档的：第一是因为三个月的redolog因为循环写问题丢失
       - 第二是redolog是物理日志 脱离了当时的物理硬盘环境 日志无法重放。逻辑操作可以在任何地方重放。

     

2. 只有binlog 没有redolog

   - binlog的缺点 不知道结果
     - 断电重启后 内存里有很多脏页 去询问binlog binlog 只知道逻辑上修改了什么 
     - 但是不知道脏页写进了硬盘里没有 
     - binlog无法记录物理页的状态 innodb没办法进行精确的崩溃恢复
     - 也就是分不清断电重启后 哪些脏页需要重新刷进去
   - redolog redolog
     - 是循环写的 有个核心概念叫做 checkpoint 
     - checkpoint之前的数据 代表刷入了 这部分的redolog空间就可以被覆盖 
     - 断电重启后 innodb会寻找最新的checkpoint 然后顺序往后执行 

### Q为什么 MySQL 既需要 Redo Log，又需要 Binlog？

A： “因为它们的职责、维度和写入方式完全不同，缺一不可。

 **Redo Log 是物理日志**，循环覆盖写，专门服务于 InnoDB 引擎的**崩溃恢复（Crash-safe）**，保证事务的持久性； **Binlog 是逻辑日志**，追加写不覆盖，专门服务于 Server 层的**数据归档恢复**和**主从复制**。 两者各司其职，共同构成了 MySQL 高可用和高可靠的坚固基石！”



## 两阶段提交

目前是mysql每次提交一个事务 都会写两份日子 一份给引擎保命，一份给从库同步。

会出现主从数据不一致的危机：如果写了redolog 刚要写binlog 就断电了。

1. 引擎拿着写好的redolog去 因为内存里确实有这个数据。

2. 但是binlog没写进去 意味着从库收不到命令 则会出现主从数据不一致的问题。

也就是断电后 从库的数据无法被恢复。



### 两阶段提交

1. prepare： 执行commit准备提交事务的时候 准备吧物理修改先写入redolog 但是会打上一个prepare标志。
2. 写binlog 此时mysql server把sql写入binlog 并且落盘保存  此时归档指令记好 随时可以发给从库
3. commit 当看到binlog写入硬盘 innodb会把prepare状态修改为commit

### crash-safe 崩溃回复

1. 如果redolog刚写完断电
   - mysql发现redolog是prepare binlog为空
   - 则从库肯定写不进去 为了主从一致 直接回滚事务【回滚事务依靠的是undolog 则主从库都没修改
2. 写完binlog 还没修改prepare为commit
   - mysql发现redolog prepare 但是binlog有
   - 此时还是会修改commit 并且提交事务 主从一致。

### 拓展 ：redolog怎么知道binlog那条和他对应

XID。

1. **Prepare：** 引擎将物理修改写入 Redo Log，打上 Prepare 标志，**同时在这个 Redo Log 里记录下当前事务的专属 `XID`**。

2. **写 Binlog：** Server 层把 SQL 写入 Binlog 落盘，**同时也会把刚才那个一模一样的 `XID` 记录到 Binlog 中**。

3. **崩溃恢复判定：** 拿着 Prepare 状态 Redo Log 里的 `XID`，去 Binlog 里搜索。如果有同样 `XID` 的完整记录，就说明 Binlog 落地成功，执行 Commit；如果没有，就调用 Undo Log 回滚。
