# 索引

1. 聚簇索引 clustered index
   - 叶子结点是完整的行数据
   - 主键ID为索引 【非叶子结点
2. 二级索引/非聚簇索引
   - create index name on user(name)
   - 为user表的name字段建立一个索引 
   - 则建立的这个B+树的排序规则是名字的字典序
   - 叶子结点存主键id
3. 回表查询 
   - select * from user where name = 'zhangsan' 
   - 先走name的二级索引b+树 然后找到zhangsan的id
   - 再回表 用id去聚簇索引B+树找出张三的完整数据
4. 覆盖索引 covering
   - select id,name from user where name='zhangsan' 不查询all 查询id and name  
   - 不产生回表 在二级索引树上直接找到了所有需要的信息
5. 联合索引/复合索引 composite index
   - create index idx_name_age on user(age,name)
   - 为age name字段都建立一个合并的索引
   - 树的排序规则 ：按照age 从小到大排序 再按照name
   - 叶子结点不止存id 是存age name id 则不会发生回表

## Q：什么是覆盖索引 怎么回表回表：

A 覆盖索引不是一种全新的索引结构 ，是一种极致的查询状态。当我们执行SELECT查询的时候 想查询的字段 包括where条件和select查询的列 刚好都在某个二级索引树的叶子结点中 mysql会直接从这个树返回所有需要的结果。

不会拿着二级索引树查出来的id 再去聚簇索引里进行第二次查找 避免了回表查询。

工程中：遇到频繁查询并且字段固定的sql 我们为这些字段建立联合索引，制造出覆盖索引的条件来提升性能。

### 联合索引不一定会产生覆盖索引，覆盖索引不一定是联合索引。

- 比如建立了联合索引 但是查*

- 索引只有一个 查的刚好都在二级树中 就发生了覆盖索引。





## 最左前缀法则 leftmost prefix rule

建立联合索引以后 二级索引树的排序规则很固定，按照属性顺序排序。没办法跳过某一个属性。

结论：

只要建立了联合索引(a,b,c) where条件必须包括最左边的字段a 才用得上这个二级索引树，直接用b和c去查，这个树相对于我 是乱序的，索引失效 退化为全表扫描。

### 特例

1. `SELECT * FROM orders WHERE A = 1 AND C = 3;`

- 部分用到了索引。按a去找 找到了a的区域。

- 但是c失效了，则退化为a范围内的线性全表扫描。

### -索引下推

- 没有索引下推的时代，会查出所有a范围的id 全部去回表，再去server判断c是否=3 这样会导致大量的回表。

- 新时代：在a的区域中 二级索引的叶子结点直接找出c=3的 符合这些的再去回表。减少了回表次数。

- “索引下推的本质，就是**将原本需要在 Server 层进行的条件过滤动作，下推（Pushdown）到了存储引擎层（InnoDB）去执行**，从而大幅减少了存储引擎将其完整数据返还给 Server 层引发的无效回表次数。”

2. WHERE b = 2 AND a = 1 AND c = 3

用得上索引 因为 MySQL 的优化器（Optimizer）非常聪明，它会在执行前自动帮你把条件重排为 `a = 1 AND b = 2 AND c = 3`，从而完美契合最左前缀法则。所以最左前缀看的是**查询条件中是否包含了最左字段**，而不是在 SQL 语句中书写的绝对先后顺序。
