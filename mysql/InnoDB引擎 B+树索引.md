# InnoDB引擎 B+树索引

DB 数据结构的选取：

数据库最大的问题是磁盘I/O次数 内存访问是纳米级别 磁盘访问是微秒。索引设计的唯一摩的就是尽可能减少磁盘I/O次数。

1. Hash
   - O1的查找时间复杂度
   - 但是不支持范围查询 会退化成全表扫描
2. 二叉查找树/红黑树
   - 支持等值查询 范围查询 时间复杂度为O logn
   - 树太高了 二叉树每个节点最多只有两个分支 树特别高 磁盘IO次数高
3. B树 多路平衡查找树
   - 每一个节点里 存索引 & 整行数据
   - 由于OS按页读取磁盘 默认16KB 一个节点一页 则一个节点自己的数据就占去大部分页内存了 剩下的内存放不了几个指针 则一个结点连不了多少的字节点
   - 这就导致树高还是变高了 因为分支减少 并且做范围查询的时候 还得在分支上上下横跳 IO次数增多

## B+树

1. 非叶子结点只存 索引【ID 不存真实数据
   - ID占的内存很小 则16KB的一页/一个节点 可以塞得下特别多指针【索引   分支就变得特别多 树就变得很扁平矮胖
2. 叶子结点使用双向链表连接 并且数据全部存放在叶子结点
   - 方便进行范围查找

### 提问 有什么B+树叶子结点的双向链表很适合范围查询

1. 如果没有链表 比如B树 查完一个id后 还得回到树顶 重新查找范围内的其他id 为进行大量无意义的随机磁盘io
2. b+树找到区间边界结点 由于叶子结点以双向链表连接 所以顺着链表横向遍历即可
3. 结合磁盘的预读机制 相邻的页面会被一次性加载到内存的buffer pool中 把非常耗时的随机io-->顺序io 范围查找性能大大提升

### 页面

innoDB引擎中 一个结点强制对应一个页面 【16KB 会塞得满满的。 都是索引

问题就是插入一个结点 则会产生页分裂， 会去申请一个新空页放该节点的信息 还回去修改树的上层的目录指针，告诉大家 位置变了。

发生页分裂 会产生空间碎片，更严重的是导致磁盘I/O飙升，写入性能暴跌。

#### UUID

所以生产中 主键采用自增的整型ID 避免随机生成的UUID 可以避免页分裂机制。自增就保证新出现的结点不会插入在之前的结点中。

还有内存问题：

自增 `BIGINT` 只需要 8 字节。

UUID 字符串通常需要 36 字节。

**核心逻辑**：主键占用空间越大 -> 非叶子节点（16KB）能放下的指针就越少 -> B+ 树的分支变少 -> **B+ 树变高** -> 磁盘 I/O 次数增加，查询性能下降
