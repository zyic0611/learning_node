# 秒杀架构

## v1版本 java的内存校验 直查mysql

- 需要判断数据库有没有库存，以及有没有下单过。
- 多线程并发情况下 多个线程同时查数据库 发现 库存>0
- 多线程同时查 订单表 发现也没有下单过
- 此时就会在java内存里先扣除 库存数量 做-1 再update回db
- 此时就会出现 **check then act** 。因为读和写没有办法保证原子性。会导致严重的超卖和一人多单
- 并且每次都直接连接mysql mysql的连接池有限 并且和硬盘交互效率低 所以响应速度慢



## v2版本  引入redis

1. 使用redis的setnx->java的setIfAbsent 保证防重刷 使用 DECR扣库存。

   - 这两个命令都是原子性操作  两个线程无法同时执行decr扣减库存 ；两个线程同时发现没有一人一单也无法同时下一人一单标记

   -  redis的命令是单线程的 

   - 但是原子操作和原子操作之间是分开的 多个原子命令组合在一起就不是原子的了 中间的网络波动/GC停顿 可能导致状态错乱。

     - 因为按理来说先在在redis里查询出由库存了 就设置一人一单 然后再redis里扣库存 数据库这时候先不减

     - 但是如果两个原子操作中间停顿 可能设置了一人一单 redis里的库存被别的人先扣走了。此时就不够原子性。

2. 网络I/O问题 一次秒杀和redis交互3次 防重查询 查库存 扣件库存 网络往返【RTT  拖慢了响应时间

3. mysql行锁问题 【最严重

   -  redis扛住了超卖  但是成功的请求仍然去同步写mysql 多个线程同时update一行数据 触发innoDB的排他行锁 大量线程再db阻塞排队 api响应时间提升

4. 丢失更新问题 check then act

   - 虽然不会超卖 但是由于在内存里修改了库存再写入db 会导致丢失更新。

   - 多个线程同时读取到stock=5 都修改为4 此时就少扣减了很多次库存。

## V3 引入lua脚本和MQ

1. lua脚本保证redis一系列操作的原子性：
   - 先限流操作：几秒内不得重复抢 查询一人一单 再查询库存>0 再标记一人一单 预扣减redis库存 
2. 引入MQ 
   - 实现消费者：service层完成了redis的一系列判断操作 直接返回成功 把生成订单 修改数据库的操作交给消费者去完成
   - 削峰填谷 真正占时间的是修改数据库操作的时间 交给mq慢慢去做 前台api只做内存扣件+投递信息 加速了api响应时间
   - 消费者必须解决check then act问题 ，使用mp的update操作 把读和写包装成原子操作。避免了超卖问题。【V2也可以使用mp的update操作 解决一下超卖。



## 并发测试数据

设置50个线程，专家号有10个。

### 如果固定是一个老人ID号

- **关于 V2 为什么没触发严重行锁：** 因为 50 个请求里有 49 个在 Redis 层面就被 `setIfAbsent` 拦截了，**真正到达 MySQL 执行 `UPDATE` 的只有 1 个线程**。既然只有 1 个线程在写数据库，就不存在“竞争（Contention）”，自然也就没有“排队等锁”的耗时，所以它本地单机跑得很快。

- **关于 V3 为什么慢几毫秒：** 除了 Lua 脚本的解析开销，更因为 V3 那唯一成功的 1 个请求，需要和 RabbitMQ 建立连接并投递消息。这种**中间件的“冷启动”和网络序列化开销**，在单机少量成功请求的测试中，会被放大。

### 如果是随机老人ID号

- **V2 的灾难：** 是**前台的 Tomcat 线程（也就是用户的 API 请求）**在死死等待行锁释放。锁等得越久，用户的接口响应时间就越长，最后直接超时报错。

- **V3 的优雅（异步解耦）：** 前台 Tomcat 线程把订单扔进 MQ 后就**立刻向前端返回成功了（耗时极短）**。真正去面对 MySQL 行锁、去排队受苦的，是**后台的 MQ 消费者线程**。消费者在后台慢慢排队写库，完全不影响前台 API 的飞速响应！这就叫**“削峰填谷”**。
