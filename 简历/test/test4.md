## 一、 Redis 底层与高并发架构 (结合医疗项目)

1. 

   **[Redis 内存淘汰策略]** 在你的智慧医疗平台中，Redis 既被用来做 Lua 脚本的库存预扣减 ，又被用来维护专家的热榜 。如果因为大促活动，海量的缓存数据瞬间把 Redis 的物理内存写满了，Redis 会直接宕机吗？为了保护核心业务，你会如何配置 Redis 的内存淘汰策略（比如 LRU 和 LFU 的区别是什么）？

   不会直接宕机。物理内存满了，会根据我们配置的内存淘汰策略来清理内存。

   由于需要保护核心业务，所以需要优先删除过期的数据。

   LRU是最近最少使用 强调离现在的时间。LFU是最少频率使用 基于长期使用的频率来统计。

   LRU会出现冷门key最近被调出使用 导致LRU成为最新的 不会被清理，而去清理相对热门的数据 ，不利于保护核心业务。

   所以应该配置内存淘汰策略为volatile-lfu.

2. 

   **[分布式 ID 生成]** 在 RabbitMQ 异步削峰的环节 ，当消费者成功拿到订单消息准备落库时，你不能再依赖单机 MySQL 的自增 ID 了（因为分库分表或多节点部署下会冲突）。在分布式高并发场景下，你会用什么方案来生成一个全局唯一、且趋势递增的订单 ID？（提示：雪花算法 Snowflake）

   分布式环境下 如果使用自增ID 很容易出现冲突。 

   雪花算法 生成64位Long数字。高度结构化。

   - 41位毫秒级时间戳 保证ID的趋势递增 避免页分裂
   - 10位的 机器ID 防止不同机器生成相同ID
   - 12位序列号 支持统一机器一ms内生成4096个不同ID

   mb框架内值了雪花算法的ID生成器 在id字段上+注解

   @TableId(type=IdType.ASSIGN_ID) 实现高性能的分布式ID分发

   

## 二、 Elasticsearch 核心机制 (简历上的亮点未发掘)

1. 

   **[ES 倒排索引原理]** 简历里提到你搭建了基于 Elasticsearch 的检索引擎，支撑海量医疗资源的筛选 。面试官最爱问：你能用大白话解释一下什么是“倒排索引”吗？如果一个患者搜索“心血管内科专家”，ES 底层是怎么把这句话拆解，并快速找到对应医生的？

   倒排索引：就是 先统计好索引表中有多少词 再把出现这些词的documentid作为一个列表 作为词的value 逻辑上是一个hash结构。通过词去找医生 。

   1. 先利用分词器 把心血管内科专家分词 
   2. 再通过倒排索引找出分词后的每一个词对应的医生id 
   3. 然后根据bm25算法 计算相关性分数 给id排序
   4. 返回排序后的医生
   5. 整个检索时间接近o1或者ologn

   

2. **[ES vs MySQL LIKE]** 同样是搜索医生名字或者科室，我完全可以在 MySQL 里用 `WHERE department LIKE '%心血管%'` 来查，为什么非要大费周章引入一个笨重的 Elasticsearch 集群？这两者在查询的底层时间复杂度上有什么本质区别？

​	使用like 的模糊查询 无法走索引 会全盘扫描 时间复杂度大

​	使用ES 只需要分词后 去倒排索引里 本质上先在内存的term index fst数据结构上找到磁盘偏移量 然后再去磁盘的term dictionary对应位置 二分查找一小段四句 然后找到对应的列表。里面就是医生的id 速度很快。

## 三、 Java 集合底层与并发 (纯纯的基本功)

1. 

   **[ConcurrentHashMap 原理]** 你的简历写了具备扎实的算法与数据结构功底 。在多线程环境下缓存某些字典数据时，我们通常不用 `HashMap` 而是用 `ConcurrentHashMap`。请问在 JDK 1.8 中，`ConcurrentHashMap` 是如何保证线程安全的同时，还能维持极高并发读写性能的？（提示：抛弃了分段锁，用了 CAS + Synchronized）。

   抛弃了分段锁 锁的细粒度进一步升级,细化到了每一个bucket的头节点。。当插入节点的时候，如果没有发生哈希冲突  采取无锁化 则直接尝试使用cas添加节点，

   如果发生了，则对链表或者红黑树头节点上synchronized锁。 其他桶的读写操作不影响

   并且读的时候  get操作无锁 因为底层的node节点值和指针都是volatile修饰  保证了多线程环境下的内存可见性 不会并发冲突 并发读效率高

## 四、 计算机网络实战 (八股文必考)

1. 

   **[TCP 可靠传输]** 在你的医疗影像异步推理闭环中，Java 端需要接收前端传来的高清医疗影像图片 。图片在网络中传输走的是 TCP 协议。请问 TCP 底层是如何保证这张好几兆的图片数据，在复杂的网络环境中传输时**绝对不丢包、不乱序、不损坏**的？（提示：序列号、确认应答、超时重传、滑动窗口）。

   1. 序列号与确认答应： tcp给发送的每一个字节都打上了序列号 由于网络路径不同 先发的数据可能后到 接收端拿到包后 在自己tcp操作系统内核层面 通过序列号凭借排序 解决了乱序问题。 并且每接收到一个包 会回复一个ack 确认答应 告诉发送端  序列号多少以前的数据我都收到了 下次从序列号+1开始发
   2. 重传机制：发送端发送图片后 迟迟没收到ack
      - 超时重传：设置定时器 时间到了没收到ACK 直接重发
      - 三次重复ACK 接收端发现少了一个包 则会连续发3个重复ACK 发送端收到就直接重发丢失的包
   3. 滑动窗口：解决了接收方接受了过多的数据 导致后面的包丢失 接收端会在返回的ACK上写上自己的缓冲区空间大小 发送端根据窗口控制自己的发送速度 流量控制
   4. 拥塞控制 ：解决网络卡顿问题 tcp会采取慢启动 拥塞避免等算法 探查网络堵塞情况 动态调整发包速度 避免包丢失
