# 支付接口

## redisson

`lock.tryLock(0, -1, TimeUnit.SECONDS)` 

第一个参数是等待时间 对于支付这件事 如果点击支付 抢不到锁 就直接不抢了 说明有一个支付的线程正在占用的锁 等待他支付完自己再去拿这个锁没意义，不需要重复支付。

第二个参数 -1 也就是允许线程无限期持有一个锁，这里是用到了redisson的看门狗机制 如果支付接口内 调用支付api 或者修改数据库啥的 卡顿 多花了点时间 允许等待。

### 不用setnx

在秒杀接口中 是使用setnx/lua基本 在redis里占个标记位防并发 ，为什么支付不用呢。

setnx会有到期时间：

1. 由于第三方延迟导致的锁提前释放：
   - 由于网络卡顿等原因，用户还没完成支付，redis自动删除锁，用户以为没反应 就再次点击支付接口，该线程也顺利拿到锁，也进去调用payapi，则会重复扣钱。

2. 会误删别人的锁

   - redis删除了锁 a线程卡顿恢复后也会继续执行本线程 继续完成支付。
   - 最严重的是线程末尾有释放锁的代码，但此时锁是b线程+的 则又会把b的锁释放掉，引发一系列循环的问题 因为此时c线程又能进来。

3. 把到期时间设置久一点不就行了吗？比如1min 或者不设置

   - 如果服务器出问题 那么用户到下一分钟才能再次发起支付

   - 不设置肯定不行 如果服务器出问题 会一直卡顿在里面 造成死锁 锁无法被释放

### redisson解决方案

1. 看门狗机制

   - 锁里面的任务结束才释放锁 完美解决上述问题1。

2. UUID+ThreadId作为value

   - 使用setnx无法 value里塞的是没有意义的1. 线程delete的时候 不知道这把锁是不是自己的。 虽然key都是订单号 同一个key。

   - redison使用 UUID+ThreadId作为value 
   - 当线程在finally释放锁的时候 必须判断一下这个锁是不是自己的value
   - 因为是根据key删除锁 这种情况下 key都是订单号 可能订单号一样 value不一样。会删除错误

3. 释放锁的check-then-act问题

   - 比如check后发现是自己的value 准备删除的时 锁刚好到期了 此时被别人抢到锁 写上了他的value 就发生了误删

   - redisson的lock.unlock封装了lua

     - 由于redisson为了支持可重入锁 底层采用的是hash 不是string。
     - hash结构为 key 锁名 value-key :uuid+ThreadId value-value 可重入次数

     - 先查hash看是不是自己的锁
     - 如果是 可重入扣减1
     - 如果可重入次数还>0 则重新刷新一下过期时间 
     - 否则释放锁

### 一人一单标记和 分布式锁的区分

1. 秒杀的一人一单就是用来记录**结果**的 只要抢到票了 就需要标记 并且这个标记不用消失 别人看到这个标志就会拦截你
2. 分布式锁是用来保护正在进行操作的排他性    保证此时此刻只有一个人在执行流程 执行完就需要释放锁 让别人来执行 。由于这个流程不一定成功，所以还要保留让别人来做这个流程的机会 
3. 也就是说 一个是结果在标记就要在 一个是流程结束就要释放锁 面向结果和面向流程+锁 
4. 分布式锁更笨重 如果给秒杀接口+redisson也可以 锁住了过程 结果肯定保证锁住了 就是性能更差一点 锁的细粒度越大 更安全 并发性能就下降了

## lua优先 redisson的原因

1. 网络 I/O 的“三步走”放大延迟

- **V3（Lua 脚本）：** 客户端拿着一段写好的脚本，发给 Redis。Redis 关起门来瞬间执行完“查限流 + 查一人一单 + 扣库存 + 盖章”，然后返回。**全过程只有 1 次网络往返（RTT）**。
- **Redisson（分布式锁）：**
  1. 第一次网络交互：向 Redis 发送一段复杂的 Lua 脚本去**抢锁**。
  2. 第二次网络交互：抢到锁后，在 Java 内存里判断一人一单，再发指令去 Redis **扣库存**。
  3. 第三次网络交互：扣完库存后，向 Redis 发送一段复杂的 Lua 脚本去**释放锁**。 **结论：网络通信开销直接放大了至少 3 倍！**

2. 将“极速的并发”强行变成了“缓慢的串行”

- **V3（Lua 脚本）：** 相当于大家在一个巨大的售票箱前，每个人伸手进去抓一张票。Redis 的单线程处理这种极其简单的指令（`DECR` 和 `SET`），一秒钟能处理 **10 万次**。
- **Redisson（分布式锁）：** 相当于在售票大厅建了一个小黑屋，每次只允许一个人进去。进去后还要反锁门，慢吞吞地核对身份、拿票、开门出来。后续的 9999 个人全都在门外排队（或者拿不到锁直接被踢掉）。这种对同一资源的加锁排队，会让单机的 **QPS 断崖式下跌**。

3. 锁的粒度思考
   - 其实并发慢 往往不是因为用了锁 而是锁的粒度太粗 导致大量线程在抢夺同一个锁
   - 支付环节 锁粒度具体到单笔orderId 大家互相不干扰 性能很高
   - 秒杀环节 粒度很粗 只能锁住doctorId 肯定会导致严重的排队

## Transactional 事务失效

首先锁必须包裹着事务 ，否则锁释放了 事务还没提交 就会出现错误。

则必须把事务 变成一个方法 有锁的事务去调用 事务方法。

但是同一个类里面 方法A调用方法B 事务就会失效。

### 代理对象proxy

Sprin管理的bean不是原生写的bean 。

类里方法如裹有事务注解，spring启动的时候会创建一个proxy代理对象 可以理解为类的替身。【Aop

serviceimpl类只会干具体的业务代码 不懂开启事务 提交事务。

当外部controller调用serviceimpl的锁方法 aop一看这个方法没有加事务注解 则不需要调用代理对象 就直接让serviceimpl自己干活 。

他自己在锁方法内部再调用事务方法 由于此时没有代理对象 则事务完全不生效。

所以就必须手动的叫出代理对象 让代理对象去调用这个事务方法。

### 其他解决方法

1. 把事务方法放在别的serviceimpl里
2. 自己注入自己 把代理对象注入进来
