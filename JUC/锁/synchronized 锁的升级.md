# synchronized 锁的升级

JDK 1.6 优化

1. **升级路径**：无锁 -> 偏向锁 -> 轻量级锁 -> 重量级锁。**（注意：锁只能升级，不能降级！）**

2. **各阶段核心特征**：

- **偏向锁**：适用于“单线程交替执行”。对象头记录 Thread ID，消除同步开销。
- **轻量级锁**：适用于“少量线程、短时间竞争”。不阻塞线程，依靠 **CAS + 自旋** 获取锁，避免操作系统用户态与内核态的切换。
- **重量级锁**：适用于“大量并发竞争”。依赖操作系统的 Mutex Lock，未抢到锁的线程全部进入**阻塞状态**（交出 CPU）。

3. **为什么要有锁升级？**

- 早期的 `synchronized` 无论竞争大小，直接就是重量级锁，太重了。锁升级的设计，是为了在**不同竞争激烈程度下，将性能损耗降到最低**。

## 具体解释

1. **偏向锁 (biased lock)** ：当线程第一次进入同步快，会在对象头里记录自己的 thread id，下次这个线程再来，发现id都一样，直接进入，都不用cas，开销很低。
2. **轻量级锁(lightweight lock)**：当有第二个线程来竞争锁，发现thread id 不是自己的，偏向锁升级成轻量级锁，线程会用cas操作去修改对象头，如果失败，自选等待，为什么？因为让线程睡觉和唤醒需要从用户态切换到内核态，太费时间了，不如让 CPU 空转等一下。
3. 重量级锁(heavyweight lock): 如果自选等待的线程，转了很多圈(默认10圈)，或者此时又过来第三个，第四个竞争者，此时升级成重量级锁，对象头直接指向底层的一个monitor对象，所有拿不到锁的线程，全部放弃cpu，进入等待队列,也就是blocked。



## 锁升级的代价 

**为什么高并发下不建议锁升级？**

- **偏向锁撤销开销大**：撤销偏向锁需要等待全局安全点（SafePoint），会引发 Stop The World (STW)，导致系统短暂卡顿。
- **自旋耗尽 CPU**：轻量级锁在极高并发下，大量线程同时自旋（空循环），会导致 CPU 负载瞬间打满。

**大厂真实调优方案（面试杀手锏）**：

- 在明确的高并发场景下，直接通过 JVM 参数 `-XX:-UseBiasedLocking` **关闭偏向锁**，避免无谓的撤销开销。
- **版本差异说明**：因为维护成本远大于收益，**JDK 15 开始官方已经默认废弃了偏向锁**。
