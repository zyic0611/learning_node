# 可重入性

## 为什么需要可重入

复杂的业务中，经常会同步方法互相调用，嵌套的情况。比如method_a内部调用method_b 。如果这两个方法需要的是同一把锁，如果线程获取了锁，进入了a方法，准备调用b方法，但是b方法被锁上了，如果锁不可重入，那么永远调用不了b方法。

**可重入性赋予了锁“识别线程身份”的能力。同一个线程可以多次获取同一把锁，只需在最终退出时保证加锁和解锁的次数匹配即可**

核心作用就是允许一个线程多次获得同一把锁，避免自己锁自己的情况发生。

## 实现

### synchronized 基于jvm的monitor对象

在 Monitor 内部，有两个极其关键的属性：

- **`_owner`**：用来存当前拿着锁的线程是谁（认人）。
- **`_recursions`**：用来存重入的次数（计数）。

**流程**：

1. 线程 A 第一次进同步块，发现 `_owner` 是空的，直接填上自己的名字，同时把 `_recursions` 变成 `1`。
2. 线程 A 在同步块里又调了另一个同步方法，Monitor 一看，`_owner` 就是 A 啊！于是放行，把 `_recursions` 变成 `2`。
3. 线程 A 每执行完一个同步方法退出时，`_recursions` 减 1。
4. **只有当 `_recursions` 减到 0 时**，Monitor 才会把 `_owner` 清空，这时候别的线程才能进来抢锁。

### reentrantlock 基于AQS架构

`ReentrantLock` 是纯 Java 代码写的，它底层依赖一个叫 **AQS（AbstractQueuedSynchronizer）** 的队列同步器。 AQS 里也有两个对应的属性：

- **`exclusiveOwnerThread`**：记录当前独占锁的线程（认人）。
- **`state`**：一个 `volatile` 修饰的 int 变量，用来计数。

**流程**：

1. 线程 A 第一次调用 `lock()`，通过 CAS 操作把 `state` 从 0 改成 1，并把 `exclusiveOwnerThread` 设为自己。
2. 线程 A 再次调用 `lock()`，发现 `exclusiveOwnerThread` 是自己，不用再做 CAS 比较了，直接把 `state` 变成 2。
3. 每次调用 `unlock()` 时，`state` 减 1。
4. **同样，只有当 `state` 减到 0 时**，才会清空 `exclusiveOwnerThread`，并唤醒排队等待的其他线程。

## 总结

1. 实现核心要素 **当前持有线程的标识** + **重入计数器**
2. 原理对比：

- **`synchronized`（隐式可重入）**：基于 JVM 层面的 ObjectMonitor 实现。内部通过 `_owner` 属性记录当前线程，通过 `_recursions` 属性记录重入次数。退出时计数器递减，归零则释放锁。
- **`ReentrantLock`（显式可重入）**：基于 JUC 的 AQS 框架实现。内部通过 `exclusiveOwnerThread` 变量记录当前线程，通过操作被 `volatile` 修饰的 `state` 变量进行加减计数。归零时唤醒同步队列中的下一个节点。

3. 注意点 

- `ReentrantLock` 是手动挡！由于它是显式的，代码里写了多少次 `lock()`，在 `finally` 块里就**必须**写多少次 `unlock()`，否则 `state` 永远退不到 0，这把锁就永远不会被释放！
