# 线程中断

## 两种停止

1. 不能使用stop 会立即释放锁 导致数据不一致 或者 	资源未释放
2. 使用interrupt() 中断机制 

- 他只是修改一个标识位 isInterrupted()为true 只是请求停止 并不是立即停止
- 如果线程正在运行，则不断去检查标识位 然后自己选 是抛出异常 还是直接return
- 如果在sleep or wait 则无法检查，就要抛出异常 把线程吓醒 在异常中决定如何退出



## interrupt的区别

**Return法**：检测到信号 -> `return;`。简单直接，像员工辞职直接走人。

**异常法**：检测到信号 -> `throw new InterruptedException()`。这更规范，像员工辞职还写了个离职报告，上层领导（调用者）能知道你是“因为中断”而停止的，而不是莫名其妙消失的。



## synchronized时interrupt

如果a线程持有重锁，b线程在blocked，让b interrupt b还是会死等，拿到锁后，才会看中断标志。

## 能否响应中断

**synchronized（不可中断锁）**：

- 一旦进入阻塞等待锁的状态，**无法被中断**。
- **风险**：如果发生死锁，线程将永远卡死，无法通过 `interrupt()` 救回。

**ReentrantLock（可中断锁）**：

- 使用 `lock.lock()`：行为和 `synchronized` 一样，不响应中断。
- 使用 `lock.lockInterruptibly()`：**响应中断**。等待时被中断会直接抛异常退出。
- 使用 `lock.tryLock(time)`：**超时自动放弃**。等不到锁，时间一到自己就走了。
